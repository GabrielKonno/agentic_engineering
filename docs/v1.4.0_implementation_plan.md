# v1.4.0 Evolution — Critical Assessment & Implementation Plan

## Context

The user designed 14 evolutions for the Agentic Engineering Framework, centered on restructuring the self-validation loop to use real subagents (Task tool) instead of inline checklists. The core thesis: the implementing agent judging its own code has confirmation bias. Independent subagents with isolated context eliminate this.

---

## Brutally Honest Assessment

### The core thesis is RIGHT

The confirmation bias problem is real. An LLM that just wrote code using pattern X has a statistical tendency to confirm pattern X is correct when reviewing its own code in the same context window. Spawning independent subagents with isolated context is the technically correct solution. This is the single most impactful improvement the framework can make.

### What's excellent in the proposal

1. **The bias analysis is precise** — mechanical vs judgment classification of each validation step is correct
2. **The "evidence package" correction** — filesystem access > data packaging. Correct.
3. **Context routing rules (ALWAYS/IF/NEVER)** — the NEVER list (no project.md, no logs, no implementation plans) is the real innovation. Anti-bias firewall.
4. **The `invocation` field** — makes the agent/skill taxonomy functional, not just organizational
5. **The orchestration sequencing** — code-reviewer → security-reviewer → Red Team → validator → Blue Team is logical and each step feeds the next
6. **The correction that validator does ALL verification** — not just judging evidence the principal curated

### What has real problems

#### Problem 1: Token cost explosion (SEVERE)

Each subagent call creates a new conversation. Estimated per-subagent cost:
- Read agent .md file: ~80-130 lines
- Read all rules files: ~50-200 lines
- Read CLAUDE.md sections: ~100 lines
- Read git diff: ~50-500 lines
- Read prior reports (for validator): ~50-200 lines
- Reasoning + report generation: ~2000-5000 tokens output

Conservative estimate: **20-40k tokens per subagent call**.

For a typical logic-heavy task:
- code-reviewer: ~25k tokens
- security-reviewer: ~30k tokens
- validator: ~40k tokens
- **Total: ~95k tokens just for validation**

Compare to current inline approach: ~5-10k tokens. That's a **5-10x increase**.

For a 5-task sprint: ~500k tokens on validation alone. This directly reduces the number of tasks per session (currently 3-5, could drop to 2-3).

**This is the single biggest risk of the entire evolution.** The document acknowledges it ("Accept as cost of unbiased validation") but doesn't quantify the impact or propose mitigation beyond "monitor."

**Recommendation:** Task complexity MUST modulate validation depth:
- **Routine tasks** (UI text, config, styling): inline checklist is fine. No subagent. The bias risk on a text change is near-zero.
- **Logic-heavy tasks**: code-reviewer subagent + validator subagent (2 calls).
- **Architecture/security tasks**: full chain (code-reviewer + security-reviewer + Red Team + validator + Blue Team).

This graduated approach preserves the token budget for tasks where bias actually matters while keeping routine tasks fast.

#### Problem 2: The arbitrator — initially assessed as over-engineering, BUT kept at bootstrap

False ❌ from a subagent (validator says ❌ but mechanical evidence says ✅) is an edge case. Initially recommended deferring the arbitrator to on-demand creation. However, the user correctly identified that **there's no persistent memory mechanism to track false ❌ frequency across sessions**. Unlike bugs (tracked in Known Bug Patterns), false ❌ occurrences would be invisible between sessions, making the reactive trigger impractical. The arbitrator stays at bootstrap — the complexity cost is manageable (one additional agent template), and the AI handles bootstrap complexity automatically.

#### Problem 3: NEVER list may be too restrictive (MEDIUM)

Blocking `project.md` from subagents prevents bias contamination, but project.md contains **architectural decisions** that affect code correctness. Example: project.md says "Decision: use integer cents for all monetary values." The code uses integer cents. The validator, not knowing this decision, might flag it as unusual.

The context routing says to include "CLAUDE.md sections: Key Patterns, Architecture" — but some architectural decisions are ONLY in project.md (especially decisions made mid-project, not at bootstrap).

**Recommendation:** Allow subagents to read project.md's **Architectural Decisions table only** — this is factual (decisions made) not reasoning (why they were made). Block the Progress Log (session entries contain implementation reasoning).

#### Problem 4: Blue Team timing change is undocumented (MINOR)

Currently Blue Team runs during Step 3 (self-review) if Red Team ran. The proposal moves it to AFTER the entire validation loop passes. This changes when security fixes are applied — they now happen during the retry loop, not during initial implementation. This is actually BETTER (Blue Team assesses final state) but the document doesn't explicitly call out that this is a deliberate timing change.

#### Problem 5: Framework doc growth (MINOR)

The framework doc is 1353 lines. Adding a Validation Orchestration Protocol section (~150-200 lines) is significant. The bootstrap prompts (1423 and 1393 lines) would each grow by ~150-200 lines for new agent templates and orchestration instructions.

This is manageable but requires disciplined writing. The new sections should replace existing content where possible, not just add on top.

### Decisions after discussion with user

| Topic | Decision | Reason |
|-------|----------|--------|
| Validation depth | **Graduated**: routine=inline, logic-heavy=2 subagents, arch/security=full chain | Token cost — preserves budget for where bias matters |
| Arbitrator | **Created at bootstrap** (user's pushback accepted) | No persistent memory for false ❌ frequency across sessions — reactive trigger is impractical |
| Context routing | **Allow Architectural Decisions table** from project.md; block Progress Log | Prevents false ❌ on code following architectural decisions |
| Scope | **All 14 evolutions + graduated validation** (15 total) | Full scope in v1.4.0 |
| Terminology | Use "implementing agent" instead of "principal" | Consistency with existing framework language |

### Key design decisions for implementation:

1. **Graduated validation depth** (maps to task complexity classification):
   ```
   Routine task (UI text, config, styling, simple CRUD)
     → inline checklist (current v1.3 behavior). No subagent.
     → Bias risk near-zero. Token cost: ~5-10k.

   Logic-heavy task (business rules, calculations, state machines, financial)
     → code-reviewer subagent + validator subagent (2 calls)
     → Token cost: ~50-65k. Acceptable for where bias matters.

   Architecture/security task (new module, cross-module, Red Team trigger)
     → full chain: code-reviewer + security-reviewer + Red Team + validator + Blue Team
     → Token cost: ~120-150k. Worth it for high-risk tasks.
   ```

2. **Validator agent template** — the most complex new component:
   - Independently re-runs build and tests (intentional redundancy)
   - Evaluates test quality (do tests actually assert what criteria describe?)
   - Executes VERIFY: criteria via browser/command
   - Executes QUERY: criteria via database
   - Decomposes multi-step criteria (Step 5a)
   - Executes mutation tests (Step 5c) for logic-heavy/arch tasks
   - Receives prior review reports (code-reviewer, security-reviewer, Red Team) as additional evidence
   - Produces structured report with ✅/❌/⏭️
   - BOUNDARIES section defines what NOT to read

3. **Arbitrator agent template** — three terminal outputs:
   - UPHOLD ❌: validator was right → implementing agent fixes, re-submits to validator (not arbitrator)
   - OVERRIDE TO ✅: validator was wrong → proceed, log justification
   - ESCALATE: genuinely ambiguous → human decides
   - Trigger: validator ❌ AND mechanical evidence contradicts
   - NOT triggered: validator ❌ AND mechanical evidence agrees (legitimate ❌)

4. **Context routing** — per subagent type:
   ```
   ALWAYS read:
     - The agent's own .md file
     - .claude/rules/*.md (ALL rules — cost low, omission risk high)
     - CLAUDE.md sections: Key Patterns, Architecture
     - project.md: Architectural Decisions table ONLY

   IF security-relevant:
     - .claude/agents/security-reviewer.md
     - Stack security skill in .claude/skills/

   IF UI task:
     - Design System section of CLAUDE.md

   NEVER read (anti-bias firewall):
     - project.md Progress Log (contains implementation reasoning)
     - .claude/logs/*.md (session history)
     - Sprint proposals or implementation plans
     - Any file the implementing agent wrote as part of the task explanation
   ```

5. **Commit flow**:
   ```
   Phase A (implementing agent):
     Implement → build → tests → commit "feat: [task] — pending validation"

   Phase B (subagents, sequential):
     code-reviewer → [security-reviewer] → [Red Team] → validator → [arbitrator] → [Blue Team]

   After validation:
     If all ✅: proceed (feat commit stands)
     If any ❌: fix → commit "fix: [task] — validation fix N" → re-spawn from code-reviewer
     Max 3 retry cycles. After limit: STOP and escalate.
   ```

6. **Orchestration sequencing rationale**:
   - Code-reviewer FIRST: catches code patterns cheaply before expensive validation
   - Security-reviewer SECOND: security findings affect ✅/❌ judgment
   - Red Team THIRD: adversarial testing before final validation
   - Validator FOURTH: receives ALL prior reports as input, does independent verification
   - Arbitrator FIFTH: only if validator ❌ contradicts mechanical evidence
   - Blue Team LAST: assesses FINAL code state (post-fixes from retry loop)
